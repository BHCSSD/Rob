# Line Following Navigation

In this lab, you will use two IR sensors pointed at the ground to detect a black line on a white surface. Your robot must interpret these "light vs. dark" readings to stay on track.

> [!IMPORTANT]
> **Reflective Logic:** IR sensors work by bouncing light off the floor.
> * **White Surfaces** reflect light (**Signal = 0 / LOW**).
> * **Black Tape** absorbs light (**Signal = 1 / HIGH**).
> * *Note: Some sensors may be inverted; always test with the Serial Monitor first!*
> 
> 

---


# Level 1: The "Digital Eyes" (Status Reporting)

Before moving the motors, your robot needs to "know" where it is. We will use the **Serial Monitor** (Ctrl+Shift+M) to debug our sensors.

## The Goal

Calibrate your sensors so they can clearly tell the difference between black tape and the white floor.

## The Logic

Program your robot to print its status based on the four possible states:

| Left Sensor | Right Sensor | Serial Message |
| --- | --- | --- |
| 0 (White) | 0 (White) | `"Off the Line"` |
| 0 (White) | 1 (Black) | `"Veering Right"` |
| 1 (Black) | 0 (White) | `"Veering Left"` |
| 1 (Black) | 1 (Black) | `"On the Line"` |

---

# Level 2: Reactive Steering

Now, translate those Serial messages into motor movements.

## The Goal

Keep the robot centered on the line by counter-steering.

## The Logic

* **Both on Black:** Drive Forward `set_speed(1, 1);`
* **Left on Black:** Robot is drifting right: **Turn Left** `set_speed(0, 1);`
* **Right on Black:** Robot is drifting left: **Turn Right** `set_speed(1, 0);`
* **Both on White:** Robot has lost the line: **Stop** `set_speed(0, 0);`

# Level 3: Smooth Navigation & Error Handling

Instead of fixing a jittery robot with complex code, you will use hardware engineering to smooth out the ride and add a safety fail-safe.

### Task A: Designing a Physical Deadband

If your sensors are too close together, they will both hit the black line at the same time, causing the robot to "stutter" or vibrate.

* **The Fix:** Adjust the width of your IR sensors so the black line fits cleanly **between** them.
* **The Result:** This creates a **Physical Deadband**. The robot only turns when a sensor "trips" over the edge of the line.

### Task B: The "Lost Robot" Timer (Fail-Safe)  

> [!WARNING]
> Skip this part
> 

If the robot misses a sharp turn and both sensors see white for too long, it should stop instead of wandering away.

**The Logic:** Use a variable to count how long the robot sees white.

```cpp
int lostTimer = 0; 

if (leftFloor == 0 && rightFloor == 0) {
    lostTimer++; // Increment the timer
    if (lostTimer > 500) { // If lost for ~1 second
        set_speed(0, 0);   // Emergency Stop
    }
} else {
    lostTimer = 0; // Reset timer because we found the line!
}

```

# Level 4: The "Search Pattern" Bonus

Before giving up and stopping, can you make your robot hunt for the line?

## The Challenge

If the line is lost (Both == White), execute a **Search Routine**:

1. **Rotate left** for 200ms.
2. **Rotate right** for 400ms.
3. **Wiggle** back and forth to rediscover the black path.

> [!TIP]
> Use the `millis()` function or a specialized counter to time these movements without stopping the whole program
